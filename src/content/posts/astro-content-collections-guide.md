很多个人博客项目会在开始时把 Markdown 文件直接扫出来使用，短期看很快，但一旦文章数量上来，元数据缺失、字段拼写不一致、页面读取逻辑分散的问题就会一起出现。对 Astro 来说，`content collections` 最大的价值不是“能读 Markdown”，而是把内容当成一种有类型约束的数据源来管理。

## 为什么我会优先定义 schema

博客文章最容易失控的是 frontmatter。有人写 `date`，有人写 `pubDate`；有人把标签写成字符串，有人写成数组；还有人会忘记补描述字段。只要 schema 不收口，这些问题最后都会落到模板层去兜底。

把 schema 提前定义好有两个直接收益：

1. 写文章时能尽早发现字段错误。
2. 列表页、详情页、归档页都能共享同一套稳定字段。

对 v1 博客来说，`title`、`description`、`pubDate`、`category` 是必填项，`updatedDate`、`tags`、`cover`、`featured`、`draft` 是可选或带默认值的扩展字段。这样既不会把写作门槛抬得过高，也能保证后续页面不需要写大量防御性判断。

## slug 为什么必须来自英文文件名

中文文件名对本地写作很友好，但直接拿来做路由会带来两个问题。第一是 URL 可复制性和可读性不稳定，第二是在 OSS 或其他静态托管环境里，路径编码问题会增加排查成本。把 slug 固定为英文 kebab-case 文件名，本质上是在给部署链路降噪。

这一层规则也让内容和路由的边界很清楚：

- 文章标题服务于阅读体验。
- 文件名服务于 URL 稳定性。

这样做之后，后续做上一篇、下一篇和分类聚合时就不需要再处理额外的 slug 覆盖逻辑。

## 页面读取逻辑应该集中而不是分散

如果首页、分类页、归档页都各自调用 `getCollection("posts")` 再自己排序和过滤，看起来没问题，但逻辑会很快复制开。更稳的做法是把这些规则统一放到一个 `content.ts` 里，例如：

- 全部文章读取与草稿过滤。
- 按发布时间降序排序。
- 精选文章选取。
- 分类聚合。
- 归档聚合。
- 上一篇与下一篇查找。

这样后续页面只消费结构化结果，不再各写一套排序和筛选。页面层只负责展示，数据层负责规则收口，这对长期维护会更省力。

## v1 阶段我刻意不做的事情

有些能力当然可以继续加，比如标签聚合、搜索索引、系列文章、阅读时长、目录锚点高亮，但这些内容一旦加入，页面和数据模型会一起变复杂。当前阶段的目标很明确：先把静态博客的主数据流做稳定。

对一个纯静态博客来说，只要这几件事可靠，站点就已经具备长期写作的基础：

- Markdown 能被稳定校验。
- 页面能从同一套内容工具读取数据。
- 列表与详情的排序规则一致。
- 构建结果不依赖运行时服务。

先把这些基础打牢，后续再补功能，成本会低很多。
