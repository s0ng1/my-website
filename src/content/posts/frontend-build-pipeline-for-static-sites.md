很多静态博客项目在开始时会被当成“随便写写的页面集合”，于是目录结构、公共数据函数和页面职责都很快混在一起。等到文章数量增加、页面增加，再回头整理时成本就会上来。对我来说，静态博客一样需要一条干净的前端构建链路，只是这条链路更偏向内容组织和模板稳定性。

## 目录结构的目标不是整齐，而是减少判断成本

我会把目录拆成几类：

- `content` 存 Markdown 文章。
- `components` 存可复用展示模块。
- `layouts` 负责页面骨架。
- `utils` 负责内容聚合和通用逻辑。
- `data` 存站点级静态配置。

这样做的意义不是看起来“规范”，而是当你修改某个需求时，能立刻判断应该去哪个层级动手。比如文章排序，一定不应该散落在多个页面模板里；它应该收敛到 `utils/content.ts`。

## 页面层要少做决定

页面模板最容易失控的一点，是一边拉数据、一边排序、一边聚合、一边处理样式。短期当然能工作，但页面一多，每个页面都会藏一份自己的规则。更合理的方式是先让数据层输出结构化结果，再让页面层只管使用。

例如首页文章区需要的是：

- 当前精选文章。
- 排除精选后的最新文章。
- 分类入口统计。

这些结果最好在进入页面前就已经准备好，而不是在 `index.astro` 里临时拼接。页面层越少做决定，后续 UI 调整越安全。

## 构建和预览应该尽早纳入日常步骤

有些项目会等功能快做完才第一次跑完整构建，这种顺序很容易把问题压到最后。静态博客反而更适合尽早把 `check`、`build`、`preview` 当成日常动作，因为：

- 页面都能在构建期暴露问题。
- Markdown schema 会直接影响生成结果。
- 路由是否完整也能很早看出来。

只要这一套流程养成习惯，新增文章和新增页面都不会变成一次高风险改动。

## 个人博客最值得优先做的工程化

我不会在 v1 就引入太多工具，但有几件事很值得尽早做：

1. 内容 schema。
2. 数据聚合工具。
3. 统一 layout。
4. 可复用卡片组件。
5. 固定的构建与预览命令。

这些工作听起来不如首页视觉显眼，却决定了博客是否能长期稳定演进。对个人项目来说，真正高价值的工程化不是复杂，而是让未来的自己少做重复劳动。
